# Admin Site-Locale Setting — Design

- **Date:** 2026-04-20
- **Status:** Approved (brainstorming → writing-plans)
- **Repo:** `nexibase/nexibase` (upstream). Downstream consumer: `nexibase/demo`.

## 1. Goal and Scope

### 1.1 Goal

Expose the existing `site_locale` setting as an admin-editable control in `/admin/settings`, so operators can change the site's display language (`ko` / `en`) through the UI — without touching the database or reinstalling. Today `site_locale` is written only during install (`src/lib/install/runInstall.ts`) and there is no post-install affordance to change it.

### 1.2 In scope

- A "Site language" dropdown added to the existing "Site basics" card on [src/app/[locale]/admin/settings/page.tsx](../../../src/app/[locale]/admin/settings/page.tsx), directly below the `site_name` input.
- `site_locale` threaded through the page's `SettingsData` state, defaults, load/save flow, and save payload.
- Dropdown options sourced from `SUPPORTED_LOCALES` (currently `['ko','en']` — auto-generated by `scripts/scan-plugins.js`). Human labels via a static map: `ko → 한국어`, `en → English`.
- Automatic full-page reload (`window.location.reload()`) after a successful save **only when** the saved locale differs from the locale that was loaded. Any other setting change saves without a reload.
- Two new i18n keys in the `admin` namespace: `siteLocale` (label) and `siteLocaleDescription` (helper text under the input). Added to both `src/messages/ko.json` and `src/messages/en.json`.

### 1.3 Out of scope (explicitly deferred)

- User-facing language switcher. Nexibase remains a single-language site per `src/i18n/request.ts`: the DB `site_locale` is authoritative; browser `Accept-Language` and URL hints are ignored. Introducing per-user locale switching would invert that policy and is a separate design.
- Changing `routing.defaultLocale` or `localePrefix`.
- Adding new locales beyond what `SUPPORTED_LOCALES` already contains. Adding locales is handled by the existing plugin-scanner pipeline and is independent of this feature.
- Locale-change audit log, confirmation modal, or preview.
- Server-push invalidation. `src/i18n/request.ts` already intentionally avoids caching (see its header comment), so no cache busting is needed.
- Changes to the install wizard. The install flow already captures a locale; this feature only adds a post-install editing path.

### 1.4 Design principles

- **Zero API churn.** `POST /api/admin/settings` already upserts arbitrary keys with no whitelist; adding a new key is a pure frontend change plus i18n strings.
- **Follow the existing settings-page pattern.** `site_locale` is one more field in the same `SettingsData` shape, the same "Site basics" card, the same load/save lifecycle.
- **Locale list stays auto-generated.** The dropdown reads `SUPPORTED_LOCALES` from `src/i18n/_generated-locales.ts` rather than hard-coding options, so future locales added by the scanner appear automatically.
- **No silent UX surprises.** Because SSR-rendered strings on the currently open page were generated in the old locale, a save that changes the locale must visibly re-render; we force a reload.

## 2. Architecture

### 2.1 Responsibility split

| Layer | Responsibility |
|---|---|
| DB | Existing `settings` table (`key` unique, `value` text). No schema change. `site_locale` row already exists from install. |
| Read path (`src/i18n/request.ts`) | Unchanged. `getSiteLocale()` returns DB value; falls back to `routing.defaultLocale` if missing. Non-cached by design. |
| Save API (`src/app/api/admin/settings/route.ts`) | Unchanged. Upserts whatever keys the client sends; admin-gated by `getAdminUser()`. |
| Settings page (`src/app/[locale]/admin/settings/page.tsx`) | New `site_locale` field in `SettingsData` and `DEFAULT_SETTINGS`; new `<Select>` UI below `site_name`; post-save reload when locale changed. |
| Locale list (`src/i18n/_generated-locales.ts`) | Unchanged. Source of truth for dropdown options. |
| Messages (`src/messages/{ko,en}.json`) | Add `admin.siteLocale` and `admin.siteLocaleDescription`. |

### 2.2 Data flow

```
Load:    page mount → GET /api/admin/settings → settings.site_locale into state
                   → localeOnLoad ref captures the value at load time

Edit:    <Select> onChange → handleChange('site_locale', value)

Save:    click Save → POST /api/admin/settings { settings: {..., site_locale} }
                   → on 200: if (settings.site_locale !== localeOnLoad) window.location.reload()
                              else show normal success toast
```

The `localeOnLoad` comparison is captured in a `useRef<string|null>`, set inside the settings-load effect once the GET response is applied to state — not from current state. That way the reload fires whenever the *persisted* locale changed since the last load, regardless of intermediate edits.

### 2.3 Components touched

Exactly one component file and one API interaction:

- **Modified:** `src/app/[locale]/admin/settings/page.tsx`
  - Add `site_locale: string` to `SettingsData` interface.
  - Add `site_locale: 'en'` to `DEFAULT_SETTINGS` (matches `routing.defaultLocale` in `src/i18n/routing.ts`). This initial value is overwritten as soon as the GET response resolves on mount, so it only affects the brief pre-load state.
  - Import `Select, SelectTrigger, SelectContent, SelectItem, SelectValue` from `@/components/ui/select`.
  - Import `SUPPORTED_LOCALES` from `@/i18n/_generated-locales`.
  - Define a component-local `LOCALE_LABELS: Record<string,string> = { ko: '한국어', en: 'English' }`. If `SUPPORTED_LOCALES` grows beyond this map, fall back to the raw locale code.
  - Add a `useRef<string | null>(null)` called `localeOnLoadRef`; set its value inside the existing settings-load effect after the GET resolves.
  - In the save handler, after a successful POST: if `settings.site_locale !== localeOnLoadRef.current`, call `window.location.reload()`; otherwise keep existing toast/state-reset behavior.
  - Insert a new `<div className="grid gap-2">` block with `<Label htmlFor="site_locale">{t('siteLocale')}</Label>`, the `<Select>`, and a `<p className="text-sm text-muted-foreground">{t('siteLocaleDescription')}</p>` — placed **immediately after** the existing `site_name` block and **before** the `site_description` block.
- **Modified:** `src/messages/ko.json` and `src/messages/en.json`
  - Add `admin.siteLocale` = `"사이트 언어"` / `"Site language"`.
  - Add `admin.siteLocaleDescription` = `"관리자 화면과 공개 사이트 전반에 적용됩니다. 저장 시 페이지가 새로고침됩니다."` / `"Applies to the admin and the public site. The page will reload when saved."`

No other files change. No API route, no Prisma schema, no install script, no i18n routing, no middleware.

## 3. UX details

- Dropdown is always visible, never hidden behind a feature flag.
- On first mount (fresh install before any save of settings), `settings.site_locale` falls back to `'ko'` from `DEFAULT_SETTINGS` so the control shows something sensible even if the GET returned no row. The first subsequent save will upsert the row.
- The reload fires only on locale change, so normal saves (e.g. editing site name) remain friction-free.
- If the user changes the locale, edits other fields, then clicks Save: the save completes, the reload fires, and the freshly reloaded page fetches from the server — so the other edits are persisted *before* the reload, no data loss.
- No confirmation modal. Locale changes are reversible in one click by re-saving.

## 4. Testing

Manual verification checklist (no automated test infrastructure exists for the admin settings page today):

1. `site_locale` shows current DB value on page load (pre-install ko demo: shows "한국어"; fresh install with `en`: shows "English").
2. Change dropdown from `한국어` → `English`, save. Page reloads. Admin sidebar and public site render English.
3. Change back to `한국어`, save. Page reloads. Korean returns.
4. Edit only `site_name` (no locale change). Save. **No reload** — staying on the same page, toast fires.
5. Admin auth check: unauthenticated POST still rejected 401 (unchanged behavior).
6. DB: `SELECT value FROM settings WHERE \`key\`='site_locale'` matches the last saved dropdown value.

## 5. Risks and mitigations

| Risk | Mitigation |
|---|---|
| User loses unsaved edits because reload fires | Reload is post-save, so all fields are persisted before reload. Still, if the save partially fails (one key errors) the current code resolves the whole `Promise.all` as rejected and the reload is skipped — verify this path in testing. |
| `SUPPORTED_LOCALES` expands and `LOCALE_LABELS` map is stale | Dropdown falls back to raw locale code (`'de'`, `'ja'`) for unmapped entries; still functional, just unpolished. Adding a label is a one-line change. |
| Admin is on a localized admin page (`/ko/admin/settings`) and switches to `en` | `localePrefix: 'never'` means there is no locale in the URL, so the reload lands on the same path and the server resolves the new locale. No redirect needed. |
| Cache staleness between save and reload | `src/i18n/request.ts` disables caching intentionally. First request after save reads new value. |
