# Vibe-Recipe Completeness — Design

**Date:** 2026-04-17
**Scope:** `src/plugins/vibe-coding-recipes/cron/prompt-builder.ts`, `src/plugins/vibe-coding-recipes/cron/recipe-validator.ts`

## Problem

Recipes generated by the `vibe-coding-recipes` plugin instruct users to create plugin/widget files that do not match the project's actual conventions. Users follow the recipe to the letter and end up with a plugin that does not appear in the registry.

Concrete failures observed in the `countdown-timer` recipe:

1. Widget `.tsx` file is created, but the required sibling `widgets/<Name>.meta.ts` file is never mentioned in any step. Without `.meta.ts`, the plugin scan cannot register the widget.
2. Widget component is scaffolded with individual props (`{ eventName, targetDate, completionMessage }`) instead of the project's convention (`{ settings }` wrapper). The widget renderer calls widgets with `{ settings: {...} }`, so individual-prop widgets receive `undefined` at runtime.
3. The `beginner widget` difficulty guide in `prompt-builder.ts` literally says "Single widget component + plugin.ts only," codifying the wrong pattern.

Root cause: the prompt that drives Claude's generation is disconnected from the actual project conventions. `WIDGET_ARCHITECTURE_CONTEXT` says widgets are "registered automatically" without mentioning the `.meta.ts` file, and no reference example is attached to anchor the output.

## Goals

- Recipes that claim to produce a widget must always include a step that creates `widgets/<Name>.meta.ts` alongside `widgets/<Name>.tsx`.
- Widget `.tsx` scaffolding in recipes must use `{ settings }` prop convention.
- Incomplete recipes are rejected at validation time so the cron regenerates instead of persisting broken output.
- No changes to cron scheduling, schema, admin UI, or existing recipe data.

## Non-goals

- Sandbox execution of generated recipes (out of scope — future work).
- Retroactive fix of already-generated broken recipes (can be regenerated manually via admin UI).
- Reshaping the JSON schema the model returns (`stepsEn`, `stepsKo`, etc. stay identical).

## Design

### 1. `prompt-builder.ts` changes

#### 1a. Expand `WIDGET_ARCHITECTURE_CONTEXT`

Replace the current two-sentence `WIDGET_ARCHITECTURE_CONTEXT` with an explicit contract:

- Widgets live as a **pair of files** in `src/plugins/<name>/widgets/`:
  - `<WidgetName>.tsx` — the React component
  - `<WidgetName>.meta.ts` — the registry entry (required, not optional)
- `.meta.ts` exports a default object with this exact shape:
  ```ts
  export default {
    title: 'Human Readable Title',
    defaultZone: 'right' | 'left' | 'header' | 'footer' | 'main',
    defaultColSpan: number,
    defaultRowSpan: number,
    settingsSchema: { /* default values keyed by setting name */ },
  }
  ```
- The widget component signature **must** be `({ settings }: { settings?: {...} })` — individual props are not supported by the renderer.
- Inside the component, destructure from `settings` with fallbacks: `const city = settings?.city ?? 'Seoul'`.

#### 1b. Attach a reference snippet

When `slot.type` is `widget` or `plugin_with_widget`, append a `## Reference Example` section to the user prompt containing:

- A compact `.meta.ts` template (6–10 lines, based on `WeatherWidget.meta.ts`).
- The first ~25 lines of `WeatherWidget.tsx` showing the `{ settings }` signature and destructuring pattern.

The reference is there to anchor the model's output, not to be copied verbatim.

#### 1c. Fix `getDifficultyGuide()` for widget beginner slot

Current text: `Single widget component + plugin.ts only / 2-3 steps`.

New text:
```
Beginner Widget:
- Three files: plugin.ts + widgets/<Name>.tsx + widgets/<Name>.meta.ts
- No schema.prisma, no admin, no API
- Self-contained UI (clock, quote rotator, counter, etc.)
- 3 steps — one per file
- Widget component MUST accept `{ settings }` prop, not individual props
- .meta.ts settingsSchema defines default values for every configurable field
```

Also update the `advanced` guide to explicitly call out the `widgets/<Name>.meta.ts` file when the plugin bundles a widget.

### 2. `recipe-validator.ts` changes

Add a new pure function `validateRecipeCompleteness(recipe: ValidatedRecipe): void`, called from `generate.ts` right after `validateRecipe()` returns.

All string matches below are case-insensitive substring checks against the concatenation of every step's `prompt` and `expected` fields in `recipe.stepsEn`.

Rules:

- If `recipe.type` ∈ `{widget, plugin_with_widget}`:
  - Must mention `.meta.ts`.
  - Must mention either `{ settings }` or `settings?:` to nudge the prop signature.
  - Failure → `throw new Error('Recipe is missing widget .meta.ts step')` or `... missing settings prop signature` — cron catches and regenerates.
- If `recipe.type` ∈ `{plugin, plugin_with_widget}`:
  - Must mention `plugin.ts` → throw if missing.
- Quality warnings (don't throw, `console.warn` in the same style as the existing step-count warning):
  - `intermediate` + `plugin` missing any of `schema.prisma`, `admin/`, `locales/`.
  - `advanced` missing `menus/` or `routes/`.

The rationale for throw-vs-warn mirrors existing validator behavior: structural defects that guarantee a broken plugin are thrown; soft quality issues are logged.

### 3. Integration point

`src/plugins/vibe-coding-recipes/cron/generate.ts` currently calls `validateRecipe(parsed)`. Add `validateRecipeCompleteness(validated)` immediately after. The cron's existing retry/log behavior handles the throw naturally.

## Testing

Manual verification:

1. Regenerate the `countdown-timer` recipe via the admin UI's "Generate" button and confirm the new recipe includes a `.meta.ts` step and uses `{ settings }` prop signature.
2. Generate one `plugin` and one `plugin_with_widget` recipe and confirm required files are referenced in steps.
3. Force a degenerate response (temporarily mock Claude to return a widget recipe without `.meta.ts`) and confirm `validateRecipeCompleteness` throws.

No automated tests — the plugin has no existing test suite for this module, and adding one is out of scope.

## Risk

- **Stricter validation causes more regenerations.** Acceptable cost; cron already retries. If regeneration rates become a problem, convert throws into warnings after gathering data.
- **Reference snippet bloats the prompt.** Adding ~40 lines of example code; well within token budget.
- **Existing broken recipes remain broken.** They will not self-heal; users must regenerate via admin UI. Acceptable — no data migration in scope.
